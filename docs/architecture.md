## 9. LXC Provisioning & Host YAML Schema

The HOST daemon provisions LXC containers using a declarative YAML file ("lxc-compose.yml" or similar) for each container. This file describes all required options for `pct create` and `pct set`, following the Proxmox documentation (https://pve.proxmox.com/pve-docs/chapter-pct.html#pct_container_images).

**YAML Structure Example:**
```yaml
vmid: 101
hostname: media
ostemplate: local:vztmpl/debian-12-standard_12.0-1_amd64.tar.zst
cores: 2
memory: 2048
swap: 512
rootfs: local-lvm:32
unprivileged: true
features:
    - nesting=1
    - fuse=1
net0: name=eth0,bridge=vmbr0,ip=dhcp,hwaddr=AA:BB:CC:DD:EE:FF
mounts:
    - mp: mp0
        source: /mnt/data
        target: /data
        options: rw
    - mp: mp1
        source: /opt/appdata/media
        target: /media
        options: rw
```

**Provisioning Logic:**
- The HOST parses the YAML and runs `pct create` with all creation-phase options (image, rootfs, cpu, memory, etc.).
- After creation, it runs `pct set` for each additional mount or post-creation option.
- The MAC address (for static DHCP) is generated by the CLIENT and written to the YAML (`hwaddr` field), ensuring collision-free but static MACs for each container.
- This approach guarantees reproducible, idempotent LXC provisioning and aligns with Proxmox best practices.

# Global Architecture & Infrastructure Decisions (Rust GitOps Refactor)

This document defines the authoritative architecture, network, and storage rules for the entire GitOps homelab, governing the 3-Tier Rust applications (CLIENT, HOST, LXC). It explicitly resolves all legacy conflicts and preserves critical infrastructure requirements.

## 1. Core GitOps & Infrastructure Philosophy
- **Git is the Absolute Single Source of Truth:** No manual `docker run` commands, and no manual file edits on the server. If it is not in the Git repository, it does not exist.
- **Idempotency & Fail-Closed Safety:** Every script, hook, and application must be able to run multiple times without causing duplicate entries or corruption. If a critical dependency fails (e.g., a missing secret, a failed mount), the system must Fail-Closed (abort and crash) rather than boot in a vulnerable state.
- **Sync Engine Conflict (Resolved):** The legacy `node-sync.sh` cron script is fully replaced by the Rust LXC Daemon, which uses an HTTP Push API for immediate syncs, with a 30-minute fallback cron job to guarantee eventual consistency.

## 2. Storage Layout & Media Rules
- **Stateless LXCs & Centralized Host Data:** Application data is never permanently stored inside the LXC container filesystem. All persistent configuration data lives on the Proxmox Host's fast NVMe SSD under `/opt/appdata/<STACK>`.
- **Unprivileged Bind Mounts:** The host directory is shared with the unprivileged LXC at `/appdata` via bind mounts. Docker Compose files reference this path as `/appdata/<STACK>/<APP>/config`.
- **Automatic Directory Creation:** The Rust scaffolding and LXC daemon ensure all required bind-mount directories are created on the Proxmox host before containers start.
- **Separation of Media Storage:** Large, replaceable media files are kept off the NVMe backup path. They are stored on separate spinning-disk arrays and mounted directly into the LXC at `/mnt/data/18TB` and `/mnt/data/12TB`.
- **Remote Path Mappings:** Remote Path Mappings are still required for the Arr suite (Sonarr, Radarr, etc.) to translate download client paths (e.g., from `/mnt/downloads`). This is a manual configuration in the Arr web UI and must be maintained for correct media import.

## 3. Networking, VPN, & Security
- **MAC-Based Static IPs:** IP addresses are never hardcoded inside the LXC OS. The CLIENT generates a safe MAC address, and the router (OPNsense) uses DHCP reservations to assign stable static IPs based on that MAC.
- **SSH Alias Management:** Access is managed via `~/.ssh/config` aliases (e.g., `ssh media`, `ssh gateway`), preventing the need to memorize IP addresses.
- **Isolated Docker Bridge Networks:** Apps that need to communicate within the same stack are placed on a shared, isolated Docker bridge network (e.g., `paperless_network`). These are created idempotently by pre-deploy hooks before Docker Compose runs.
- **VPN Kill-Switch (Network Namespaces):** Containers that require VPN protection (like qBittorrent) must use `network_mode: service:gluetun` to share the network stack. If the VPN drops, the app loses all internet access instantly. This logic is enforced in all generated compose files and documented for all VPN-dependent apps.
- **Proxy & Security Conflict (Resolved):** Nginx Proxy Manager is entirely deprecated. The architecture now uses Traefik with the CrowdSec L7 Bouncer middleware for all reverse proxying and security. All stacks must use Traefik labels and CrowdSec middleware; Nginx Proxy Manager is not to be deployed or referenced anywhere in the system.

## 4. Secret Management Paradigm
- **No Encrypted Commits (Goodbye SOPS/Age):** Storing encrypted secrets in Git (via SOPS) is fully deprecated in favor of dynamic runtime injection.
- **Ephemeral Secrets Container:** The LXC daemon spins up a temporary Docker container prior to deployment to fetch secrets from a secure vault, write them to a local `.env` file (`chmod 600`, `.gitignore`d), and immediately exit. If this fails, the deployment halts. Infisical and pre-sync.sh are no longer used for secrets.

## 5. Centralized Observability & Telemetry
- **Universal Log Aggregation:** Every stack includes a Promtail container that ships all Docker logs to a centralized Loki instance on the monitoring LXC.
- **Structured `logfmt` Telemetry:** The LXC GitOps Engine emits structured logs (`ts=... level=... stack=... app=... msg=...`). Promtail parses these to extract `level`, `stack`, and `app` as queryable Loki labels.
- **Grafana Auto-Provisioning:** Grafana is auto-provisioned directly from Git, with Loki as a datasource. The "Homelab Logs" dashboard dynamically discovers newly deployed stacks and apps based on Loki labels, requiring zero manual dashboard edits when adding a new app. This remains a core architectural feature.

## 6. System-Wide Security Integrations
- **Traefik + CrowdSec (L7 Bouncer):** Traefik serves as the GitOps-driven reverse proxy and integrates the official CrowdSec Bouncer middleware to analyze access logs and block malicious IPs across the entire homelab. Nginx Proxy Manager and its CrowdSec integration are fully removed.
- **Automated OS Security Patching:** During LXC provisioning, the HOST daemon automatically installs `unattended-upgrades`. This guarantees that the underlying Debian/Ubuntu OS in every container receives critical security patches without manual intervention.

## 7. CI/CD & Monorepo Workflows
- **Monorepo Path Filtering:** The entire 3-Tier architecture lives in a single Git repository. GitHub Actions uses path filters (`paths: 'client-app/**'`) so that changes to one tier do not trigger builds for another.
- **Automated Release & Self-Healing:**
    *   The **LXC Daemon** is built as a Docker image, pushed to GHCR, and automatically updated across all containers via Watchtower.
    *   The **HOST Daemon** compiles natively via GitHub Actions, publishes to GitHub Releases, and pulls its own updates automatically.

## 8. Terminal UI/UX (TUI Conflict Resolved)
- **Gum is Fully Removed:** All terminal interfaces globally use Ratatui for a consistent, premium TUI/UX. Gum is not used anywhere in the system. All interactive management, dashboards, and modals are implemented in Rust using Ratatui, with a unified color palette and animated spinners.
