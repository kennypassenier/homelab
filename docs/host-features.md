## LXC Provisioning from YAML

- The HOST daemon reads a declarative YAML file ("lxc-compose.yml") for each container, describing all options for `pct create` and `pct set`.
- The daemon applies creation-phase options first, then sets additional mounts and features post-creation.
- The MAC address is always set from the YAML, as generated by the CLIENT, to ensure static, collision-free network identity.



# Tier 2: HOST (Proxmox Host Daemon)

The HOST is a native `systemd` daemon running directly on the Proxmox bare-metal hypervisor, fully replacing the old `host.sh` and all legacy management scripts. All logic is implemented in Rust. **host.sh is deprecated and must not be used.**

**Communication model:** The HOST daemon only responds to requests from the CLIENT. It does **not** call LXC daemons directly. All orchestration between HOST and LXC flows through the CLIENT, which waits for HOST confirmation before issuing any follow-up call to an LXC.

## 1. Plain Output (No Ratatui)
- [ ] **Rudimentary Logging Only:** The HOST daemon emits plain `stdout`/`stderr` log lines in logfmt format. There is no Ratatui TUI. All interactive management and visual feedback lives exclusively in the CLIENT.
- [ ] **Structured Logfmt:** Every event follows `ts=<ISO8601> level=<info|warn|error> component=host stack=<name> msg="..."`. Log lines are the sole output surface.
- [ ] **SSE Stream:** All log events are also broadcast via `GET /api/events/stream` so the CLIENT can render them live in the deployment modal.

## 2. Infrastructure Provisioning
- [ ] **Proxmox API Client:** Uses `reqwest` to interact with the Proxmox API. It clones Debian templates to create new LXCs and injects the MAC addresses generated by the CLIENT (replacing the old `bootstrap-lxc.sh` flow).
- [ ] **Post-Provisioning Exec Hook:** Immediately after LXC creation, the HOST daemon uses the Proxmox Exec API to atomically run `apt-get update && apt-get upgrade`, install Docker ([7]), and install `unattended-upgrades` for automatic OS security patching ([8], [9]). If the `apt` command fails, the HOST daemon implements a Fail-Safe and deletes the broken LXC immediately. The HOST emits a completion or failure event; the CLIENT waits for this event before issuing any call to the new LXC daemon.

## 3. Atomic Hardware Passthrough
- [ ] **GPU & TUN Parsing:** Replaces `enable-gpu.sh` and `enable-tun.sh` ([10], [11]). The daemon reads `/etc/pve/lxc/<vmid>.conf` to configure hardware acceleration (e.g., for Jellyfin [12]) or VPN tunnels (e.g., for Gluetun [13]).
- [ ] **Atomic Rename:** To prevent file corruption, it writes the new `lxc.cgroup2.devices.allow` rules to a temporary file, then performs an atomic `rename` operation to overwrite the config.


## 4. Backup Orchestration & Stack Recovery
- [x] **API-Driven Restic Backups:** Replaces `backup-stacks.sh` ([14]). The CLIENT triggers backup via `POST /api/backup/run`. The HOST runs Restic on `/opt/appdata` and streams progress via SSE. The HOST does **not** contact LXC daemons directly — the CLIENT is responsible for calling `POST /api/backup/pause` on each LXC before triggering the HOST backup, and `POST /api/backup/resume` after it completes.
- [x] **Rust Drop Guards:** A Rust Drop Guard on the HOST guarantees that a failure event is emitted to the CLIENT even if the Restic process panics, so the CLIENT can still issue resume calls to paused LXCs.
- [ ] **Persistent Storage Management:** The HOST daemon is responsible for managing the actual NVMe storage at `/opt/appdata/<STACK>`. It relies on the CLIENT to scaffold and create the required directory structures, but performs atomic wipes of these directories when a reset-stack action is triggered, ensuring safe and complete data removal while preserving the LXC container itself.

## 5. CI/CD Self-Updater
- [ ] When changes are pushed to the `host-daemon/` folder, GitHub Actions compiles the Rust binary and publishes it to GitHub Releases.
- [ ] The HOST daemon periodically checks the GitHub Releases API, downloads the new binary to `/tmp/`, atomically overwrites its current executable, and restarts itself via `systemctl`.
